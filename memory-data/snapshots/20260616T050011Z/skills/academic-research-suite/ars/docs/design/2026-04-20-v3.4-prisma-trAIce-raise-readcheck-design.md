## 1. Background

ARS v3.3.6 shipped on 2026-04-15. Three gaps flagged during the NYCU EMBA fact-check exercise:

1. **A. PRISMA-trAIce** 鈥?2025 reporting checklist for AI-assisted systematic reviews (Holst et al. 2025, *JMIR AI*, doi:10.2196/80247). 17 items across 6 sections, 4 priority tiers (Mandatory / Highly Recommended / Recommended / Optional). GitHub canonical: `cqh4046/PRISMA-trAIce`.
2. **B. RAISE** 鈥?Cochrane / Campbell / JBI / EPPI-Centre / CEE joint framework for Responsible AI in Evidence Synthesis (Thomas et al. 2025, NIHR ESG Working Group). Role-based ecosystem, not an item checklist. 8 roles 脳 2鈥? responsibilities, organised around four principles: human oversight, transparency, reproducibility, fit-for-purpose. Source: `eppi.ioe.ac.uk/CMS/Portals/0/RAISE ESG BPWG.pdf` (accessed 2026-04-20).
3. **C. Reading-check probe** 鈥?Socratic-mentor nudge to detect whether the user has actually read the AI-cited material.

v3.3 anti-sycophancy and S2 verification already cover ~80% of EMBA speaker warning points. v3.4 closes the reporting-framework gap.

### 2.2 Why mode-aware scope

- **PRISMA-trAIce** is a systematic-review reporting standard. Running it on primary research is a category mistake.
- **RAISE principles** are universal AI ethics for evidence work. Running them on non-SR is a principled extension, not official compliance.

### 3.3 Design principles

1. **Single compliance agent.** Not two agents. `compliance_mode` parameter dispatches payload.
2. **No new stage.** 2.5 / 4.5 are existing Integrity Gates. Compliance is a payload extension, not a structural change. Fail-loop and resume reuse existing machinery.
3. **Scope honesty.** `raise_framework.md` opens with a disclaimer: RAISE's official scope is evidence synthesis; non-SR use is principle extension, not RAISE compliance.
4. **Reading-check decoupled.** v3.5.0 has zero dependency on compliance_agent; ships and rolls back independently.

**Self-checks** (compliance agent's own failure modes)

| Check | What | On fail |
|---|---|---|
| CA-1 | Am I quoting PRISMA-trAIce from memory or from protocol? | Force re-read `shared/prisma_trAIce_protocol.md` |

| CA-3 | Did all 17 items pass? (Sycophancy risk) | Force second pass on 3 Mandatory items; require explicit evidence citation |
| CA-4 | Do RAISE principle passes have evidence citations? | Downgrade no-evidence passes to warn with `[weak evidence]` tag |

### 4.3 `shared/raise_framework.md`

**Structure**

1. **Scope disclaimer (must be first, non-removable):**
   > RAISE's official scope is evidence synthesis (systematic review, meta-analysis, scoping review, rapid review, evidence mapping). When ARS applies RAISE principles to non-SR work (`academic-paper full`, primary research, etc.), this is a **principle extension**. ARS does not claim official RAISE compliance for non-SR outputs.

2. **Four principles** 鈥?human oversight, transparency, reproducibility, fit-for-purpose. Per-principle: definition, ARS check procedure, pass/warn/fail criteria.

3. **Full 8-role matrix** (SR mode only) 鈥?8 roles 脳 2鈥? responsibilities each. Default ARS-user role assumption: Evidence Synthesists. ARS itself also occupies AI Development Team + Methodologist roles; this dual-role declaration is stated in the doc.

4. **Citation:** Thomas J, Flemyng E, Noel-Storr A, et al. *Responsible Use of AI in Evidence Synthesis (RAISE).* NIHR ESG Best Practice Working Group in AI/Automation, EPPI-Centre, UCL Social Research Institute. 17 July 2025. `https://eppi.ioe.ac.uk/CMS/Portals/0/RAISE%20ESG%20BPWG.pdf`

### 4.4 `shared/compliance_checkpoint_protocol.md`

**Dual-gate behaviour**

| Aspect | Stage 2.5 | Stage 4.5 |
|---|---|---|
| PRISMA-trAIce scope | M1鈥揗10 | T1, A1, I1, R1, R2, D1, D2 |
| RAISE scope | 4 principles (focus: oversight, fit-for-purpose) | 4 principles (focus: transparency, reproducibility) + 8-role matrix (SR only) |
| Block condition | SR mode AND any Mandatory PRISMA-trAIce item = FAIL | Same |
| Warn condition | Highly Recommended fail OR RAISE principle warn | Same |
| Info condition | Recommended / Optional fail | Same |
| Non-SR mode | RAISE principles only, warn-only (never block) | Same |
| Checkpoint display | FULL checkpoint + compliance section | FULL checkpoint + compliance section |

**User action UX**

- Block 鈫?list Mandatory gaps 鈫?user choice: [backfill / return to Stage 2 / acknowledge limitation (own responsibility)]
- Warn 鈫?list HR/RAISE warns 鈫?user choice: [fix / skip]
- Info 鈫?dashboard bullet only, no flow interruption

**Fail loop** reuses `pipeline_state_machine.md 搂Integrity Check FAIL Loop`: after 3 unresolved rounds, user picks [manual handling / remove AI usage / continue with "partially non-compliant" warning]. See 搂6.3.

## 6. Error Handling

### 6.1 Error taxonomy

| # | Error | Trigger | Handling |
|---|---|---|---|
| E1 | Missing material | compliance_agent needs M4 (Input Data) but passport.bibliography is empty | Apply Anti-Leakage Protocol: mark `[MATERIAL GAP: M4 鈥?bibliography not provided]`, no hallucination. Item auto-FAILs, tier dictates block/warn |
| E2 | Mode mismatch | `academic-paper full` passes `compliance_mode=systematic_review` by mistake | compliance_agent entry-point validates mode vs skill context; on mismatch returns `{decision: abort, reason: "mode/context mismatch"}`, orchestrator re-evaluates mode |
| E3 | Block loop > 3 rounds | Stage 2.5 FAIL 鈫?backfill 鈫?FAIL 鈫?鈥?exceeds 3 | Existing `pipeline_state_machine 搂Integrity Check FAIL Loop`: list unresolvable items; user picks [manual handling / remove AI usage / continue with "partially non-compliant" warning] |
# REDACTED: sensitive-looking memory line
| E5 | Schema validation fail | compliance_report does not match Schema 12 | Pipeline halts; orchestrator reports internal error; NOT appended to compliance_history (avoid polluting audit trail) |
| E6 | Upstream reference drift | `prisma_trAIce_protocol.md` snapshot_date diverges from GitHub canonical | No pipeline block; compliance_agent adds `upstream_sync_status: "stale"` to report; weekly CI `check_prisma_trAIce_freshness.py` emits warning annotation |

### 6.2 E4 user override design

Override is not "skip" 鈥?it is "leave a trace." 3-round friction ladder:

| Round | Behaviour |
|---|---|
| 1st block | User sees warning, must acknowledge; rationale string optional |
| 2nd FAIL | User must provide rationale (any length); system notes "second override attempt" |
| 3rd FAIL | User must provide rationale 鈮?100 chars; only then released |

Generated `disclosure_addendum` is **auto-injected** into final manuscript's AI disclosure section. Non-removable. This is the concrete form of the `no detection evasion` iron rule.

```yaml
disclosure_addendum: |
# REDACTED: sensitive-looking memory line
  - PRISMA-trAIce M4 (Input Data) not fully reported; rationale: <user text>
  - RAISE principle: transparency 鈥?partial; rationale: <user text>
```

### 6.3 Boundary behaviour across skills

When compliance_agent is invoked outside the pipeline (e.g., `academic-paper full` standalone):

- Does not access pipeline state machine
- Writes lightweight local state to `./compliance_reports/<timestamp>.yaml`
- Does not trigger pipeline checkpoint dashboard
- If skill has no finalize stage, hooks before `academic-paper disclosure` mode entry

### 6.4 Resume-time error cases

| Case | Behaviour |
|---|---|
| Loaded passport predates Schema 12 | Auto-migrate: append empty `compliance_history=[]`, warn user |
| compliance_history shows unresolved Stage 2.5 block | Do not auto-retry; display dashboard, let user choose next step |
| Upstream PRISMA-trAIce updated during pause | E6 path; do not interrupt resume |

### 6.5 What does NOT change

- Existing Stage 2.5/4.5 Integrity Check FAIL Loop
- Existing `pipeline_state_machine.md` exception handling
- v3.2 Failure Mode Checklist runs in parallel with compliance, scopes non-overlapping: failure-mode checks research validity; compliance checks reporting transparency
