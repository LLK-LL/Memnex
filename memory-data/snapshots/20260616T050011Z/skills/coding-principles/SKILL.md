# Coding Principles

在开始任何编码工作前，先按下面五条原则约束自己的行为。不要把这些原则当成建议；把它们当成默认执行规范。

## 1. 编码前思考

不要假设。不要隐藏困惑。呈现权衡。

LLM 经常默默选择一种解释然后执行。这个原则强制明确推理：

- 明确说明假设 — 如果不确定，询问而不是猜测
- 呈现多种解释 — 当存在歧义时，不要默默选择
- 适时提出异议 — 如果存在更简单的方法，说出来
- 困惑时停下来 — 指出不清楚的地方并要求澄清

## 2. 简洁优先

用最少的代码解决问题。不要过度推测。

对抗过度工程的倾向：

- 不要添加要求之外的功能
- 不要为一次性代码创建抽象
- 不要添加未要求的"灵活性"或"可配置性"
- 不要为不可能发生的场景做错误处理
- 如果 200 行代码可以写成 50 行，重写它

检验标准： 资深工程师会觉得这过于复杂吗？如果是，简化。

## 3. 精准修改

只碰必须碰的。只清理自己造成的混乱。

编辑现有代码时：

- 不要"改进"相邻的代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你更倾向于不同的写法
- 如果注意到无关的死代码，提一下 —— 不要删除它

当你的改动产生孤儿代码时：

- 删除因你的改动而变得无用的导入/变量/函数
- 不要删除预先存在的死代码，除非被要求

检验标准： 每一行修改都应该能直接追溯到用户的请求。

## 4. 目标驱动执行

定义成功标准。循环验证直到达成。

将指令式任务转化为可验证的目标：

- 不要这样做...	转化为...
- "添加验证"	"为无效输入编写测试，然后让它们通过"
- "修复 bug"	"编写重现 bug 的测试，然后让它通过"
- "重构 X"	"确保重构前后测试都能通过"

对于多步骤任务，说明一个简短的计划：

1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]

强有力的成功标准让 LLM 能够独立循环执行。弱标准（"让它工作"）需要不断澄清。

## 5. Git 自动留痕

当本 skill 被用于编写、修改、重构或调试代码时，默认在当前工作路径维护一个本地 Git 历史，并在完成验证后自动提交本次代码改动。

执行规则：

- 执行 Git 操作前，先确认当前工作路径是用户任务对应的项目根目录；如果路径明显不是项目目录，先询问用户。
- 如果当前工作路径不是 Git 仓库，先运行 `git init` 创建本地仓库。
- 提交前只暂存本次任务相关的代码文件，不要使用无范围的 `git add .`。
- 仅提交代码与代码测试文件，例如常见的 `.py`、`.js`、`.ts`、`.tsx`、`.jsx`、`.java`、`.go`、`.rs`、`.cpp`、`.c`、`.h`、`.hpp`、`.cs`、`.php`、`.rb`、`.swift`、`.kt`、`.scala`、`.sh`、`.ps1`、`.sql`、`.html`、`.css`、`.scss`、`.vue`、`.svelte`。
- 排除非代码文件，例如图片、视频、音频、压缩包、二进制产物、日志、缓存、数据集、模型权重、临时文件、构建输出、导出的文档和用户资料文件。
- 如果配置文件或清单文件是代码改动的必要组成部分（例如 `package.json`、`pyproject.toml`、`Cargo.toml`、`go.mod`、测试配置），可以提交；否则不要提交。
- 不要提交与本次任务无关的既有改动。若工作区已有他人或用户的未提交改动，保持隔离，只提交自己本轮修改的代码文件。
- 提交信息使用简短、明确的动宾结构，说明完成的代码目标，例如 `fix: handle empty input` 或 `feat: add retry logic`。

检验标准： `git status` 中本次相关代码改动已经提交；未被要求提交的非代码文件、生成物和无关改动仍保持未暂存或未提交状态。
